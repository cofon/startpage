# 添加网站流程大修/重构方案

## 一、现状分析

### 🎯 1. 添加网站入口统计

| 入口 | 位置 | 功能 | 可输入字段 |
|------|------|------|-----------|
| **起始页 --add** | `src/components/AddWebsitePanel.vue` | 添加单个网站 | name, url, title, description, iconData, iconGenerateData, tags, isMarked, isActive, isHidden |
| **起始页 --import** | `src/components/CommandSettings.vue` + `src/utils/indexedDB.js` | 批量导入 | JSON 文件中的所有字段 |

### 🔍 3. 现有问题

#### ❌ 代码重复
- 起始页和插件各自有独立的表单验证逻辑
- 数据标准化分散在多个地方（websiteNormalizer.js、popup.js、AddWebsitePanel.vue）
- 图标生成函数未共享（起始页有 generateDefaultIcon，插件没有）

#### ❌ 字段处理不一致
- **iconGenerateData**: 起始页自动生成，插件无法使用
- **tags**: 起始页有标签选择器，插件只能手动输入
- **markOrder**: 起始页由 createWebsiteObject 处理 (`isMarked ? markOrder || 0 : 0`)，插件直接设为 0
- **name**: 起始页自动提取域名，插件需要手动输入或自动获取

#### ❌ 验证规则不统一
- 起始页：验证 URL 格式，验证 name/title/description至少有一个
- 插件：验证 URL 格式，验证 name/title/description至少有一个
- 导入：起始页直接导入，插件会智能补全缺失字段

#### ❌ 数据补全能力差异
- 起始页：无法跨域获取元数据
- 插件：可以利用 extension API 无跨域限制获取 title/description/iconData

## 二、重构目标

### ✅ 核心原则

1. **最大代码复用**：公共逻辑放在起始页，插件通过消息调用
2. **统一数据入口**：所有添加操作最终调用统一的 API
3. **智能字段补全**：利用插件优势自动补全缺失字段
4. **严格验证规则**：统一的字段验证标准
5. **清晰的职责划分**：UI 层、业务逻辑层、数据层分离

### 📦 1. 创建 WebsiteMetadataService

**位置**: `src/services/websiteMetadataService.js`

```javascript
/**
 * 网站元数据服务
 * 提供统一的网站数据验证、标准化、补全功能
 */

import { 
  isValidUrl, 
  extractSiteNameFromUrl, 
  generateDefaultIcon 
} from '../utils/websiteUtils'

import { createWebsiteObject } from '../utils/websiteNormalizer'

/**
 * 验证网站数据
 * @param {Object} data - 待验证的网站数据
 * @returns {Object} { valid: boolean, errors: string[] }
 */
export function validateWebsite(data) {
  const errors = []
  
  // 1. URL 验证（必填）
  if (!data.url) {
    errors.push('URL 为必填字段')
  } else if (!isValidUrl(data.url)) {
    errors.push('URL 格式不正确')
  }
  
  // 2. name/title/description至少有一个不为空
  const hasName = data.name && data.name.trim() !== ''
  const hasTitle = data.title && data.title.trim() !== ''
  const hasDescription = data.description && data.description.trim() !== ''
  
  if (!hasName && !hasTitle && !hasDescription) {
    errors.push('网站名称、标题或描述必须填写至少一项')
  }
  
  // 3. iconData/iconGenerateData至少有一个不为空
  const hasIconData = data.iconData && data.iconData.length > 0
  const hasIconGenerateData = data.iconGenerateData && data.iconGenerateData.length > 0
  
  if (!hasIconData && !hasIconGenerateData) {
    errors.push('网站图标数据或 SVG 图标必须提供至少一项')
  }
  
  return {
    valid: errors.length === 0,
    errors
  }
}

/**
 * 标准化网站数据
 * @param {Object} data - 原始网站数据
 * @returns {Object} 标准化的网站对象
 */
export function normalizeWebsiteData(data) {
  // 1. 如果 name 为空，从 URL 提取
  if (!data.name && data.url) {
    data.name = extractSiteNameFromUrl(data.url)
  }
  
  // 2. 如果 tags 为空，添加默认标签 'new'
  if (!data.tags || !Array.isArray(data.tags) || data.tags.length === 0) {
    data.tags = ['new']
  }
  
  // 3. 处理 markOrder
  if (!data.markOrder) {
    data.markOrder = data.isMarked ? 0 : 0
  }
  
  // 4. 如果没有 iconGenerateData，自动生成
  if ((!data.iconData || !data.iconData.startsWith('data:image')) && !data.iconGenerateData) {
    if (data.name) {
      const svgIcon = generateDefaultIcon(data.name)
      data.iconGenerateData = encodeSvg(svgIcon)
    }
  }
  
  // 5. 使用 createWebsiteObject 创建标准对象
  return createWebsiteObject({
    ...data,
    visitCount: 0,
    createdAt: new Date(),
    updatedAt: new Date()
  })
}

/**
 * 编码 SVG 为 Base64
 */
function encodeSvg(svg) {
  if (!svg) return ''
  if (svg.startsWith('data:image/svg+xml;base64,') || svg.startsWith('data:image/svg+xml;utf8,')) {
    return svg
  }
  const encodedSvg = encodeURIComponent(svg)
    .replace(/%([0-9A-F]{2})/g, (match, p1) => String.fromCharCode('0x' + p1))
  return `data:image/svg+xml;base64,${btoa(encodedSvg)}`
}

/**
 * 批量补全网站元数据（仅插件可用）
 * @param {Array} websites - 网站数组
 * @param {Function} progressCallback - 进度回调函数 (processed, total)
 * @returns {Promise<Array>} 补全后的网站数组
 */
export async function batchEnrichMetadata(websites, progressCallback) {
  const total = websites.length
  let processed = 0
  
  for (let i = 0; i < websites.length; i++) {
    const website = websites[i]
    
    // 检查是否需要补全
    if (!website.title || !website.description || !website.iconData) {
      try {
        // 使用插件的 background.js 获取元数据
        const metadata = await chrome.runtime.sendMessage({
          action: 'FETCH_METADATA',
          url: website.url,
          fromCurrentTab: false
        })
        
        if (metadata) {
          // 只补全缺失的字段
          if (!website.title && metadata.title) {
            website.title = metadata.title
          }
          if (!website.description && metadata.description) {
            website.description = metadata.description
          }
          if (!website.iconData && metadata.iconData) {
            website.iconData = metadata.iconData
          }
        } else {
          console.warn(`[MetadataService] ⚠ 无法获取元数据：${website.url}`)
        }
      } catch (error) {
        // 网络错误、SSL 问题等不处理，只记录日志
        console.warn(`[MetadataService] ⚠ 获取元数据失败 (${website.url}): ${error.message}`)
      }
    }
    
    processed++
    if (progressCallback) {
      progressCallback(processed, total)
    }
  }
  
  return websites
}

export default {
  validateWebsite,
  normalizeWebsiteData,
  batchEnrichMetadata
}
```

import { ref, watch } from 'vue'
import { useWebsiteStore } from '../stores/website'
import { useNotificationStore } from '../stores/notification'
// 新增：使用统一的元数据服务
import websiteMetadataService from '../services/websiteMetadataService'

const emit = defineEmits(['close'])
const websiteStore = useWebsiteStore()

// 表单数据
const formData = ref({
  name: '',
  title: '',
  url: '',
  description: '',
  tags: '',
  isMarked: false,
  isActive: true,
  isHidden: false,
  iconData: '',
  iconGenerateData: ''
})

// URL 验证状态
const urlValidation = ref({ isValid: false, message: '' })

// 处理 URL 变化
function handleUrlChange() {
  const url = formData.value.url
  
  if (!url) {
    urlValidation.value = { isValid: false, message: '' }
    return
  }
  
  if (websiteMetadataService.validateWebsite({ url }).valid) {
    urlValidation.value = { isValid: true, message: '' }
    
    // 自动填充 name
    const siteName = websiteMetadataService.extractSiteNameFromUrl(url)
    formData.value.name = siteName
    
    // 自动生成 SVG 图标
    const svgIcon = websiteMetadataService.generateDefaultIcon(siteName)
    formData.value.iconGenerateData = websiteMetadataService.encodeSvg(svgIcon)
  } else {
    urlValidation.value = { isValid: false, message: '请输入有效的 URL' }
  }
}

// 监听 URL 变化
watch(() => formData.value.url, handleUrlChange)

// 提交表单
async function handleSubmit() {
  // 1. 准备数据
  const websiteData = {
    name: formData.value.name,
    title: formData.value.title,
    url: formData.value.url,
    description: formData.value.description,
    tags: formData.value.tags.split(',').map(t => t.trim()).filter(t => t),
    isMarked: formData.value.isMarked,
    isActive: formData.value.isActive,
    isHidden: formData.value.isHidden,
    iconData: formData.value.iconData,
    iconGenerateData: formData.value.iconGenerateData
  }
  
  // 2. 验证数据
  const validation = websiteMetadataService.validateWebsite(websiteData)
  if (!validation.valid) {
    notificationStore.warning(validation.errors.join(','))
    return
  }
  
  // 3. 标准化数据
  const normalizedData = websiteMetadataService.normalizeWebsiteData(websiteData)
  
  // 4. 添加到 store（会自动保存到数据库）
  await websiteStore.addWebsite(normalizedData)
  
  notificationStore.success('网站添加成功！')
  emit('close')
}
</script>

### 🎨 6. SVG 图标生成优化（简化版）

**位置**: `src/utils/websiteUtils.js`

```
/**
 * 生成默认 SVG 图标（简化版本，无缓存）
 * @param {string} name - 网站名称或域名
 * @returns {string} SVG 字符串
 */
export function generateDefaultIcon(name) {
  // 从 URL 提取域名（如果有协议前缀）
  let displayName = name
  try {
    if (name.startsWith('http://') || name.startsWith('https://')) {
      const urlObj = new URL(name)
      displayName = urlObj.hostname.replace(/^www\./, '')
    }
  } catch (e) {
    // 不是 URL 则直接使用 name
  }
  
  // 取首字母大写
  const firstLetter = displayName.charAt(0).toUpperCase()
  
  // 根据域名哈希选择背景色（避免相同首字母无法区分）
  const colorPalette = [
    '#FF6B6B', '#4ECDC4', '#45B7D1', '#96CEB4', 
    '#FFEAA7', '#DDA0DD', '#98D8C8', '#F7DC6F'
  ]
  
  let hash = 0
  for (let i = 0; i < displayName.length; i++) {
    hash = ((hash << 5) - hash) + displayName.charCodeAt(i)
    hash |= 0
  }
  const backgroundColor = colorPalette[Math.abs(hash) % colorPalette.length]
  
  // 生成SVG（带背景色区分）
  return `
    <svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 100 100">
      <rect width="100" height="100" fill="${backgroundColor}"/>
      <text x="50" y="70" font-size="50" text-anchor="middle" 
            fill="white" font-weight="bold">${firstLetter}</text>
    </svg>
  `
}
```

**说明**:
- ✅ **无需缓存**: 每次需要时直接生成，任务完成后立即释放内存
- ✅ **基于域名**: 同一域名的不同路径生成相同 SVG（如 `baidu.com/page/1` 和 `baidu.com/page/2`）
- ✅ **颜色区分**: 使用域名哈希选择背景色，避免 `github` 和 `google` 都是 `g` 无法区分
- ✅ **简单直接**: 不引入复杂的缓存机制，符合起始页低频修改的特点

## 八、关键问题解答

### Q1: 插件可以复用起始页的哪些代码？

**✅ 可以复用:**
- `websiteMetadataService.js` - 通过 content.js 桥接调用
- `websiteNormalizer.js` - 纯函数，可直接 import
- `websiteUtils.js` - 工具函数，可直接 import
- `StartPageAPI` 暴露的所有方法

**❌ 不可复用:**
- Vue 组件（AddWebsitePanel.vue 等）- 插件是原生 JS
- Store（Pinia）- 插件没有 Vue 环境
- DOM 操作相关的代码

### Q2: 导出数据时要不要导出 ID？

**建议：不导出 ID**

**理由:**
1. ID 是自增主键，导入时会重新生成
2. 避免 ID 冲突（如果目标数据库已有数据）
3. 保持数据迁移的灵活性

**导出内容:**
```
{
  version: 1,
  exportDate: '2024-01-01T00:00:00.000Z',
  websites: [
    {
      // 用户字段 + 业务字段
      name, title, url, description,
      iconData, iconGenerateData, tags,
      isMarked, isActive, isHidden,
      markOrder, visitCount, lastVisited
      // ❌ 不包含：id, createdAt, updatedAt
    }
  ]
}
```

### Q3: markOrder 的处理逻辑？

**当前逻辑** (来自 createWebsiteObject):
```javascript
markOrder: data.isMarked ? (data.markOrder || 0) : 0
```

**解释:**
- 如果 `isMarked === true`，使用传入的 markOrder 或默认为 0
- 如果 `isMarked === false`，强制设为 0（因为不需要排序）

**建议优化:**
在批量导入时，如果要保持原有顺序，应该：
```
// 按原始 markOrder 排序
websites.sort((a, b) => a.markOrder - b.markOrder)

// 重新分配连续的 order
websites.forEach((site, index) => {
  if (site.isMarked) {
    site.markOrder = index
  }
})
```

## 十、设计原则与简化策略

### 🎯 核心设计理念

**本项目为起始页应用，数据修改频率极低，应采用"功能优先、简化设计"的原则：**

1. **低频场景不优化**
   - 批量导入仅在首次部署或数据恢复时使用
   - 单个网站添加无性能瓶颈
   - 无需引入复杂的缓存机制

2. **内存资源即时释放**
   - 批量任务完成后立即释放内存
   - 不使用持久化缓存（localStorage/IndexedDB）
   - 避免增加不必要的存储复杂度

3. **直接查询数据库**
   - 单个网站添加时直接读取数据库判断重复
   - 不预先缓存图标或 SVG 数据
   - 保持数据一致性和实时性

4. **SVG 生成策略**
   - 每次需要时实时生成（基于域名哈希 + 颜色区分）
   - 同一域名不同路径生成相同 SVG
   - 不同域名即使首字母相同也能通过背景色区分

5. **Icon 获取策略**
   - 批量导入时逐个获取，失败只记录日志
   - 不实现重试队列或延迟加载
   - 国内网站无法访问时接受降级方案（使用 iconGenerateData）

### ⚙️ 技术选型说明

| 技术方案 | 是否采用 | 原因 |
|---------|---------|------|
| 内存缓存 Map | ❌ 不采用 | 批量任务少，占用内存无意义 |
| localStorage 持久化 | ❌ 不采用 | 增加存储复杂度，管理困难 |
| IndexedDB 图标缓存表 | ❌ 不采用 | 查询效率低，不如直接存 website 表 |
| 实时生成SVG | ✅ 采用 | 简单直接，无额外依赖 |
| 域名哈希去重 | ✅ 采用 | 避免同域名重复生成不同 SVG |
| 背景色区分 | ✅ 采用 | 解决首字母冲突问题 |

### D. 关键决策记录

#### 决策 1: 为什么不实现缓存机制？
**背景**: SVG 和 Icon 是否需要缓存以优化性能？  
**决策**: 不实现任何形式的缓存（内存/持久化）  
**理由**:
- 起始页项目数据修改频率极低
- 批量导入场景罕见（仅首次部署或数据恢复）
- 缓存会增加代码复杂度和存储管理成本
- 性能提升微乎其微（可能节省几百毫秒）
- 违背"功能优先、简化设计"原则

**后果**:
- ✅ 代码更简洁，易于维护
- ✅ 无缓存一致性问题
- ❌ 每次生成SVG 都需要计算（但耗时 < 1ms）
- ❌ 批量导入时可能重复获取同一域名的 icon（但概率极低）

#### 决策 2: SVG 生成基于域名而非完整 URL
**背景**: `https://www.baidu.com/page/1` 和 `https://www.baidu.com/page/2` 应该生成相同 SVG 吗？  
**决策**: 是的，应该相同  
**理由**:
- 用户感知中这两个 URL 属于同一个网站
- 避免 Marked List 中出现多个相似但图标不同的 baidu 条目
- 符合直觉和实际使用场景

**实现**:
```
// 从 URL 提取主域名
const domain = new URL(url).hostname.replace(/^www\./, '')
// 使用 domain 生成SVG，而非完整 url
```

#### 决策 3: 使用背景色区分相同首字母的域名
**背景**: github.com 和 google.com 都以 'g' 开头，如何区分？  
**决策**: 根据域名哈希值选择不同背景色  
**理由**:
- 仅靠首字母无法区分大量网站
- 背景色可以提供额外的视觉识别维度
- 哈希算法保证同一域名始终使用相同颜色

**实现**:
```
const hash = simpleHash(domain) // 计算域名哈希
const colorIndex = Math.abs(hash) % colorPalette.length
const backgroundColor = colorPalette[colorIndex]
```

#### 决策 4: 验证规则 vs 补全策略分离
**背景**: 添加网站时 iconData/iconGenerateData至少一个即可，但补全时应该如何处理？  
**决策**: 
- **验证时**: 两者至少有一个不为空
- **补全时**: 优先补真 iconData，iconGenerateData 作为降级备用

**理由**:
- iconData 是真实的网站 favicon，视觉效果更好
- iconGenerateData 是本地生成的 SVG，用于兜底
- 有 iconData 时可以不生成 iconGenerateData（节省存储）
